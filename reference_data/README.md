# Reference Data Cache

Cached game data that doesn't change frequently — pulled from authoritative sources so Claude doesn't have to re-fetch the same information across sessions or for different characters.

**This directory is gitignored except for this README** (see `.gitignore`). The data itself is regenerable and may include user-specific notes, so it stays local. This README is committed so future Claude sessions know what should be here and how to set it up.

---

## Setup (Run Once When Cloning the Suite)

If you're a new Claude session or a freshly cloned repo, run these setup steps:

### 1. Create the directory structure

```bash
cd poe_mcp_suite
mkdir -p reference_data/eldritch_implicits
```

### 2. Clone GGG's official data exports

Both of these are official Grinding Gear Games repositories — the most authoritative source for tree data, updated each league.

```bash
cd reference_data
git clone https://github.com/grindinggear/skilltree-export.git skilltree
git clone https://github.com/grindinggear/atlastree-export.git atlastree
```

### 3. Generate Eldritch implicit reference files

These are generated by fetching the poewiki.net pages and extracting the mod tables. See the "How to regenerate Eldritch reference files" section below.

## What Lives Here

**Cache (rarely changes, safe to keep):**
- `eldritch_implicits/searing_exarch.md` — Searing Exarch Eldritch implicit pool, by slot and tier
- `eldritch_implicits/eater_of_worlds.md` — Eater of Worlds Eldritch implicit pool, by slot and tier
- `shrines.md` — Shrine buff details, Screams of the Desiccated mechanics
- `skilltree/` — GGG official passive skill tree JSON exports
- `atlastree/` — GGG official atlas tree JSON exports
- `leagues/{league-name}.md` — Per-league mechanic summaries (drop tables, scarab/atlas-node lookup, unique items, strategic notes). One file per league. Current: `leagues/mirage.md` (3.28).
- `poe_api.md` — PoE Developer API reference (endpoint paths, schemas, the `children` pattern for special stash tab types, legacy endpoint inference). Source: pathofexile.com/developer/docs/reference. Critical for fixing/extending the MCP servers.

**What does NOT live here (always fetch live):**
- Item prices (poe.ninja, trade) — changes every minute
- Player-specific data (characters, stash) — fetch via PoE API
- Active league mechanics and rewards — refer to live wiki
- Trade listings — always pull from `mcp__poe__search_trade`

## File Format Conventions

Each manually-curated file should have YAML frontmatter:

```yaml
---
source: https://www.poewiki.net/wiki/... or https://github.com/grindinggear/...
fetched: 2026-05-21
league: 3.28 Mirage
---
```

This makes staleness easy to check. The GGG repo subdirectories have their own git history — `git log` inside `skilltree/` or `atlastree/` tells you the last sync.

## Refresh Policy

| Trigger | Action |
|---|---|
| **New PoE league launches** | `git pull` in `skilltree/` and `atlastree/`; re-fetch Eldritch pages |
| **GGG balance patch** | Same as above |
| **Claude suspects data may be stale** | Check frontmatter dates; re-fetch if older than current league |
| **User contradicts the cache** | Fetch from source first, then update the cache |

## Trust Hierarchy (For Claude)

When researching PoE data, prefer sources in this order:

1. **Live game state via the user** (item pastes, in-game observations, character API) — most authoritative for the user's specific situation
2. **GGG official exports** (`skilltree/`, `atlastree/`) — authoritative for game data structure
3. **Live PoB TCP connection** — calc results for the user's loaded build
4. **Live wiki fetch** via `mcp__poemcp__fetch_wiki_page` — current truth for mechanics and item pools
5. **This cache** (`reference_data/`) — fast lookup, but verify if anything seems off
6. **PoB's bundled tree data** (in the PathOfBuilding submodule) — derived from #2, usually fine for tree queries
7. **Claude's training data** — last resort, often outdated for PoE specifics

---

## Lookup Procedure for Claude

1. **Before quoting mod ranges, tree node stats, or game pools** — check if a reference file covers it
2. **If covered** — use the cached data and cite the source
3. **If not covered or potentially stale** — fetch from source, then write the extracted data here for next time
4. **If the user contradicts the cache** — fetch fresh, update the cache, and apologize for the bad info

## How to Regenerate Eldritch Reference Files

If the Eldritch files don't exist or look stale (different league than current):

```bash
# Fetch the wiki pages via the POEMCP server tool
# Or use curl + python parsing
```

In Claude session:
1. Call `mcp__poemcp__fetch_wiki_page` for these URLs:
   - https://www.poewiki.net/wiki/List_of_Searing_Exarch_implicit_modifiers
   - https://www.poewiki.net/wiki/List_of_Eater_of_Worlds_implicit_modifiers
2. The result will be saved to a tool-results file (path returned)
3. Use Python to parse the JSON result, extract mod families by slot and tier
4. Write structured markdown files to `eldritch_implicits/`

The extraction script is in this session's history if needed — basic pattern is to grep for `EldritchImplicit\d+` lines, parse out the mod family name, slot, and stat text.

## Working With GGG Tree Data

### skilltree-export
- **What it has:** Passive skill tree nodes, positions, connections, mastery effects, ascendancy nodes
- **When to use:** Cross-validation against PoB's bundled tree data; tree node lookups when PoB isn't connected
- **Format:** JSON files in `data/` subdirectory
- **Already covered by PoB?** Yes — PoB ingests this same data. The GGG copy is mostly insurance.

### atlastree-export
- **What it has:** Atlas passive tree nodes, positions, connections, atlas-specific mechanics
- **When to use:** Any atlas-related analysis or recommendations (currently no MCP tools cover this — major gap to fill)
- **Format:** JSON files in `data/` subdirectory
- **Already covered by PoB?** **No.** PoB doesn't model atlas. This data is the foundation for any future atlas planning tools.
